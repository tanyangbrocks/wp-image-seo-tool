# WP Image SEO Tool 實作進度

*最後更新：2026-07-12*

## 目前狀態

單一靜態 HTML 工具（`index.html`）核心功能已完成：使用者上傳圖片 → 自動 OCR 辨識圖片裡已存在的文字，連同**位置、大小、實際顏色**一起還原成疊在圖片上的真 HTML 文字（可被選取/索引）→ 使用者只需填 Alt 描述 → 即時（不用按鈕）產生完整可貼進 WordPress「自訂 HTML」區塊的內容。預設用免費離線的 Tesseract.js；「進階設定」支援使用者自己的 Google Cloud Vision API 金鑰提升辨識準確度，失敗時自動 fallback 回離線引擎，金鑰只存在使用者自己瀏覽器的 localStorage，不會寫進輸出的 HTML 或 commit 進 repo。已加上信心分數過濾，解決真實 AI 生圖（含照片紋理背景）誤把雜訊辨識成大量假文字行的問題。

GitHub repo 已建立並 push，Vercel 部署使用者已自行連接（Import from GitHub，Project Name `wp-image-seo-tool`，Framework Preset `Other`，Root Directory `./`，零建置流程），已確認正式環境正常運作。使用者提出三個改善建議：語言選擇選單、Alt 改成按鈕觸發+WordPress 渲染預覽、要不要自建 AI 模型——前兩項已實作，第三項已提供不建議自建模型的分析（現有 Google Vision 選項已是更划算的「換更強模型」槓桿）。

| 語言選擇選單 + 按鈕觸發產生 HTML + WordPress 渲染預覽 | `index.html` | 使用者提三個建議，實作前兩個：**(1) 語言選單**：新增可展開的語言勾選清單（繁中/簡中/英/日/韓/俄/德/法/西，預設勾繁中+英），`getSelectedLanguages()` 動態組出 Tesseract 語言字串（如 `eng+jpn`）與 Google Vision 的 `languageHints`（如 `['en','ja']`，兩套引擎的語言代碼系統不同，各自對照）；至少要勾一種語言才能上傳圖片，沒勾時擋在上傳前並顯示錯誤訊息，不會浪費一次辨識。**(2) 按鈕觸發 + 預覽**：Alt 欄位不再邊打邊即時產生 HTML（原本用 `input` 事件），改成輸入時只切換「產生 HTML」按鈕的 disabled 狀態，按下去才真正產生；產生的同時新增一個 `<iframe srcdoc="...">` 直接渲染**產生出來的那段 HTML 字串本身**（不是套用內部即時預覽狀態模擬出來的），如果 HTML 產生邏輯本身有跳脫/格式錯誤，這裡會如實呈現出來，比原本的即時疊字預覽更貼近「WordPress 貼上去後真正長怎樣」。**驗證方式**：`DataTransfer` 模擬上傳，確認語言清單渲染出全部 9 個選項且預設狀態正確；取消全部語言勾選後上傳圖片，確認被攔下並顯示錯誤訊息；重新勾選後確認 `getSelectedLanguages()` 正確反映勾選狀態；上傳測試圖後確認 Alt 打字只切換按鈕狀態、不會自動出 HTML；按下按鈕後確認 `htmlOutput`/`wpPreviewFrame.srcdoc` 都正確產生，且**直接讀取 iframe 的 `contentDocument`** 確認裡面的 `<img alt>`／疊字文字／顏色都跟預期一致（不是只檢查字串長度，是真的解析渲染結果）；OCR 辨識/信心過濾/顏色萃取邏輯本身沒有改動，用先前的測試圖回歸測試結果不變。**已知環境限制**：這次自動化測試環境 `document.hasFocus()` 是 `false`（瀏覽器分頁沒有真正的系統焦點），導致複製按鈕的 `navigator.clipboard.writeText()` 拋出 `NotAllowedError`——這是瀏覽器對非使用者主動觸發情境的標準安全限制，不是這次改動造成的迴歸（複製按鈕程式碼本身完全沒有改動，先前 session 已用真實點擊驗證過可以正常運作）。 |
| 修正真實照片背景誤辨識成大量假文字行 | `index.html` | 使用者用真實 AI 生圖（濃濃本家茶包海報，照片背景+少量文字）測試，回報偵測到 41 行文字、疊字疊得亂七八糟，畫面上全是「他 ol NoNG HoME」「xo ᴎ ヽy ᴎ」這種亂碼。**根因**：Tesseract 對照片紋理（木紋、布紋、食物表面反光）會信心滿滿地「辨識」出根本不存在的文字，因為它預設把整張圖當文件處理，沒有機制分辨「這裡真的有文字」還是「這裡只是視覺雜訊」。**診斷過程**：做了一張還原情境的測試圖（雜訊紋理背景+一行乾淨的真實文字），量測 `data.lines[].confidence`（0-100 信心分數），真文字拿到 88 分，26 個雜訊誤判全部落在 0-43 分——兩者之間有清楚的分隔帶。**修法**：新增 `MIN_LINE_CONFIDENCE = 60` 常數，Tesseract 路徑的行結果先過濾信心分數才進下游流程，低於門檻直接丟棄。**驗證方式**：同一張測試圖重跑，41→1 行（只剩真的那行），文字內容正確；另外拿先前驗證過的乾淨測試圖（黑白字、彩色字）重新回歸測試，結果跟修正前一模一樣，沒有誤殺真文字。**已知取捨**：60 分是憑一組測試資料抓出來的門檻，不是理論上的絕對值——如果之後有真實案例回報「明明有文字卻沒偵測到」，可能是文字本身很小/字體很潦草導致信心分數偏低，那時候要往下調整這個門檻，而不是整個拿掉這層過濾（拿掉會回到雜訊亂入的老問題）。Google Vision 路徑目前沒有套用同樣的信心過濾（該引擎本身對雜訊誤判的抵抗力好很多，且目前沒有真金鑰可以實測其信心分數欄位格式，先維持原樣）。 |
| 建立核心工具：OCR 文字位置/顏色還原 + Alt 驅動的即時 HTML 產生器 | `index.html` | 上傳圖片後用 Tesseract.js（`chi_tra+eng`）辨識文字內容與每一行的邊界框；用 Otsu 閾值法把框內像素分兩群（面積較小的一群視為文字本身），取該群**原始 RGB 平均色**還原文字真實顏色（不是只判斷黑白）；位置/大小換算成相對圖片 natural width/height 的百分比，字級用 CSS `container-type: inline-size` + `cqw` 單位維持縮放時的正確比例。Alt 欄位留空時不產生輸出，一填就即時（`input` 事件，非按鈕觸發）重新產生完整 HTML，複製貼上即可用。**進階設定**：使用者可選填自己的 Google Cloud Vision API 金鑰改走雲端辨識（`groupWordsIntoLines()` 把 Vision 回傳的逐字座標依 Y 座標鄰近度群組成行，模擬 Tesseract 原生的行結構，兩引擎共用同一套下游位置/顏色處理），金鑰錯誤或請求失敗時自動 fallback 回離線引擎；「記住金鑰」勾選後存本機 localStorage，不會外流。**驗證方式**：preview 工具用 `DataTransfer` + 手動 `dispatchEvent('change')` 模擬真實檔案上傳（原生檔案選擇對話框無法被自動化工具觸發），先後測試黑白文字圖與彩色文字圖（藍底紅字），確認辨識文字、位置百分比、顏色萃取（`rgb(230,40,40)` 原色抓出 `rgb(192,44,70)`，同色系）、Alt 即時產生/清空即消失、HTML 特殊字元跳脫（`"`/`<`/`>`/`&`）、複製按鈕都正確；用假金鑰測試確認 Google Vision 請求真的送達 Google 伺服器（收到結構化 400，非網路層級失敗）且正確 fallback，行為跟原本離線模式一致（回歸測試）。 |
| 建立專案治理檔案 + GitHub repo + Vercel 部署 | `CLAUDE.md`、`.gitignore`、`docs/archive-done.ps1`、`實作進度.md` | 依照 Portfolio / SkillCreatorUE5 兩個既有專案的工作慣例補齊：`CLAUDE.md` 記錄「單檔架構是刻意選擇、什麼情況才該拆檔」的判斷原則、🔴 API 金鑰絕對不能寫死進程式碼的安全規則（這個 repo 跟 Vercel 部署都公開）、核心 OCR/顏色萃取邏輯摘要、`DataTransfer` 測試技巧；`.gitignore` 排除 OS/編輯器暫存檔、`.vercel`、未來若加建置工具需要的 `node_modules`/`.env*`；`docs/archive-done.ps1` 從 Portfolio 專案原封不動複製（腳本本身完全通用，不綁定特定專案路徑）。新建 GitHub repo `tanyangbrocks/wp-image-seo-tool`，本地 `C:\wp-image-seo-tool\` push 上去；使用者自行完成 Vercel Import from GitHub 設定。 |
