拉高）；`html-builder.js` 把 `<figcaption>` 移到定位用的內層 `<div>` 外面，`<figure>` 只當外層包裝，不再是定位容器本身。**驗證方式**：修完重跑同一組合成測試，`previewCanvasWrap` 高度／最終 HTML 裡定位容器的高度都跟圖片實際顯示高度**完全相等**（改之前分別差 沒辦法顯示合併後文字裡的換行）。**自動合併相近 OCR 行**：新增 `js/line-merge.js` 的 `mergeCloseLines()`，OCR 完成後、渲染之前呼叫——把偵測到的文字行依 `topPct` 排序，相鄰兩行若垂直間距小於兩行平均高度的 50%、且水平範圍重疊至少 40%，判定是同一段落，合併成一個方塊（文字用 `\n` 接起來、bounding box - [ ] 使用者實機用真實（非合成）行銷圖驗證文字對齊修復後的效果，尤其是原本就有多行文字、字體較特殊的圖——這次用 `canvas.fillText()` 合成測試圖精確驗證了座標轉換的兩個 bug 都修好，但沒有測過真實世界字體差異造成的視覺誤差（不同字體的字寬/字高比例跟疊字用的通用字體不同，理論上仍會有一定程度的視覺落差，不是這次能一併解決的範圍）
- [ ] 使用者實機驗證「自動合併相近 OCR 行」的合併門檻（垂直間距 <50% 平均行高、水平重疊 ≥40%）在真實圖片上是否恰當——目前只用合成測試圖驗證邏輯本身正確，真實 OCR 偵測到的行距/對齊誤差可能需要微調 `js/line-merge.js` 裡的兩個門檻常數
- [ ] **使用者實機驗證捲動動畫的實際流暢度**——區塊淡出/淡入（`IntersectionObserver`）跟頂/底端橡皮筋回彈的底層 JS 判斷邏輯都已用合成事件驗證正確，但動畫本身（CSS transition 播放、`IntersectionObserver` 在真實捲動下是否確實觸發）沒辦法在這次的測試瀏覽器工具裡直接看到——這個環境的分頁完全不跑 `requestAnimationFrame`，屬於環境限制不是程式碼問題，但仍需要真瀏覽器滑一次確認手感（尤其是觸控裝置的回彈阻力/死區數值 `RESISTANCE=0.4`／`DEAD_ZONE=10px` 是否需要微調）
